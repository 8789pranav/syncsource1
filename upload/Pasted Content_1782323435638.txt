Yes, now we should design onboarding properly like an **industry HRMS product**, not like only a checklist page.

Based on Zoho People, Keka, greytHR, and Darwinbox patterns, the real onboarding module should cover: **candidate onboarding flow, employee onboarding flow, custom workflows, task/checklist automation, documents, custom forms, e-sign, offer/letter handling, stakeholder tasks, payroll/attendance/asset setup, joining-date activation, and progress tracking**. Zoho People supports customizable candidate onboarding flows, candidate details submission, documents, NDA/background authorization, e-sign, checklists, and candidate-to-employee conversion. Keka highlights digital offer letters, pre-boarding tasks, day-one checklists, document collection, e-signatures, and integrations with payroll, attendance, and asset management. greytHR supports joining workflows with reviewer task lists, attached documents, custom forms, and activation on joining date. ([Zoho Corporation][1])

# Final Onboarding Module Structure

Use this main menu:

```text
Onboarding
 ├── Dashboard
 ├── Kanban Board
 ├── New Hire List
 ├── Initiate Onboarding
 ├── Tasks
 ├── Documents
 ├── Forms
 ├── Offer & Letters
 ├── Setup
 │    ├── IT Setup
 │    ├── Asset Setup
 │    ├── Payroll Setup
 │    └── Attendance / Leave Setup
 ├── Approvals
 ├── Reports
 └── Settings
```

This is cleaner than adding too many menus. The **Kanban Board** is the main working area, **New Hire List** is the table view, **Initiate Onboarding** starts the process, and **Settings** controls all workflows.

# 1. Main Logic

Your onboarding module should be based on this model:

```text
Onboarding Workflow
        ↓
Workflow Stages / Kanban Columns
        ↓
Task Templates
        ↓
Forms + Documents + Offer + Approval
        ↓
Onboarding Instance for one candidate
        ↓
Candidate Portal + HR Kanban
        ↓
IT / Admin / Payroll / Manager Tasks
        ↓
Convert Candidate to Employee
        ↓
Post-Joining Tasks
        ↓
Complete Onboarding
```

Example:

```text
Workflow: India Full-Time Employee Onboarding

Candidate: Rahul Sharma

Stages:
Invite Sent → Candidate Action Pending → HR Verification → Pre-Joining Setup → Joining Day → Post-Joining → Completed

Tasks:
Candidate uploads documents
HR verifies documents
IT creates email
Admin assigns laptop
Payroll verifies bank details
Manager schedules induction
HR marks joined
System converts candidate to employee
```

# 3. Dynamic Kanban Board

## Where

```text
Onboarding → Kanban Board
```

This should be the main HR screen.

Keka describes onboarding stages like **pre-offer, offer release, post-offer acceptance, pre-joining, joining day, and post-joining**, so your Kanban should follow a clean stage-based model. ([Keka HR & Payroll Software][2])

## Recommended Kanban Columns

Do not make 20 columns. Keep these practical:

```text
Draft
Invite Sent
Candidate Action Pending
HR Verification
Offer / Letter Pending
Pre-Joining Setup
Joining Day
Post-Joining
Completed
On Hold
Dropped
```

## What Each Column Means

### Draft

Onboarding is created but not started.

Used when HR is still filling details.

### Invite Sent

Candidate portal invite has been sent.

Candidate has not completed forms/documents yet.

### Candidate Action Pending

Candidate needs to complete something.

Examples:

```text
Fill joining form
Upload documents
Accept offer
Sign NDA
Submit bank details
Submit statutory details
```

### HR Verification

HR verifies candidate-submitted data.

Examples:

```text
Verify PAN/Aadhaar
Verify bank details
Verify education documents
Verify previous employment documents
Approve/reject forms
Ask correction
```

### Offer / Letter Pending

Offer/appointment/NDA is pending generation, approval, sending, acceptance, or e-sign.

Darwinbox highlights digital offer letters with custom approval flows, version tracking, and digital sign-off. ([Darwinbox][3])

### Pre-Joining Setup

Internal teams prepare before joining.

Examples:

```text
IT creates email/login
Admin prepares assets
Payroll prepares salary/statutory profile
Manager prepares induction
HR prepares welcome kit
```

### Joining Day

Tasks for day one.

Examples:

```text
Mark joined
Activate login
Issue ID card
Give laptop
Complete joining confirmation
Conduct HR orientation
```

### Post-Joining

After joining tasks.

Examples:

```text
First week check-in
Manager feedback
Training assignment
Probation setup
30/60/90 day tasks
```

### Completed

All mandatory onboarding tasks are complete.

### On Hold

Candidate joining is paused.

### Dropped

Candidate rejected offer, no-show, or HR cancelled onboarding.

# 5. Kanban Drag-and-Drop Rules

Drag-and-drop should not be free. It should validate business rules.

Examples:

```text
Cannot move to HR Verification if candidate form is not submitted.
Cannot move to Offer Accepted if offer is not accepted.
Cannot move to Joining Day if mandatory documents are pending.
Cannot move to Completed if mandatory tasks are pending.
Cannot convert to employee if required employee fields are missing.
Cannot activate payroll if salary structure or bank details are missing.
Cannot activate attendance if shift/attendance policy is missing.
Cannot activate leave if leave rule is not assigned.
```

## Stage Movement Options

```text
Manual Move
Auto Move
Manual + Auto
Locked Stage
Approval Required to Move
```

Example:

```text
Candidate Action Pending → HR Verification

Auto move when:
- Candidate form submitted
- Mandatory documents uploaded
- Offer accepted, if configured
```


# 41. Backend Flow

## From Recruitment

```text
1. Candidate marked as selected.
2. HR clicks Initiate Onboarding.
3. Candidate data is pulled from ATS.
4. System suggests workflow based on entity/location/department/employee type.
5. HR reviews workflow, documents, forms, tasks, offer template.
6. Onboarding instance is created.
7. Candidate invite is sent.
8. Candidate completes portal tasks.
9. HR verifies forms/documents.
10. Offer/appointment/e-sign process is completed.
11. IT/Admin/Payroll/Manager tasks are created.
12. Joining day arrives.
13. HR marks candidate joined.
14. System converts candidate to employee.
15. Employee code, login, leave, attendance, shift, payroll, role are assigned.
16. Post-joining tasks continue.
17. Onboarding is completed.
```

## Manual HR Initiation

```text
1. HR opens Initiate Onboarding.
2. HR enters candidate and job details.
3. System suggests workflow.
4. HR starts onboarding.
5. Candidate receives portal invite.
6. Rest of the flow follows the workflow.
```

## Bulk Onboarding

```text
1. HR downloads Excel template.
2. HR uploads candidate list.
3. System validates data.
4. HR maps workflow by entity/department/employee type.
5. System creates onboarding cases in bulk.
6. Invites are sent in bulk.
7. HR tracks all candidates in Kanban.
```

# 42. Critical Validations

```text
Cannot send invite without candidate email.
Cannot move stage if mandatory tasks are incomplete.
Cannot verify document without uploaded file.
Cannot convert to employee without mandatory employee fields.
Cannot activate login before joining date, unless allowed.
Cannot activate payroll without salary structure and bank details.
Cannot assign leave without leave rule.
Cannot assign attendance without attendance policy and shift.
Cannot complete onboarding if mandatory forms/documents/tasks are pending.
Cannot delete published workflow if onboarding cases exist.
Cannot edit published workflow directly; create new version.
```
