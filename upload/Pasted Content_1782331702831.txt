Yes. Now final onboarding design should be **entity-wise configurable**, where every entity/company can have its own:

```text
Kanban board
Workflow
Candidate form
Document templates
Email templates
Checklist
Verification rules
Approval rules
Candidate portal settings
Employee conversion settings
Logs and audit
```

And if entity has no custom setup, it should use **Tenant Default**.

# Final Onboarding Module Structure

```text
Onboarding
 ├── Dashboard
 ├── Candidates
 │    ├── Add Candidate
 │    ├── Bulk Upload
 │    └── Bulk Update
 ├── Kanban Board
 ├── Document Library
 ├── Workflow Configuration
 ├── Email Templates
 ├── Checklists
 ├── Logs
 └── Settings
```

The main rule is:

```text
Reusable things are created separately.

But workflow configuration connects everything step-wise.
```

Example:

```text
Document Library = Create offer letter / appointment letter / NDA templates
Email Templates = Create candidate invite / reminder / offer email templates
Checklists = Create HR / IT / Payroll / Manager task checklist
Kanban Board = Create stages
Workflow Configuration = Decide which form, email, document, checklist, approval, verification runs at each stage
Settings = Decide entity-wise defaults and module-level behavior
```

# 2. Configuration Fallback Priority

When candidate is added, system should find configuration in this order:

```text
1. Candidate-specific configuration
2. Work mode + employee type + department + location configuration
3. Employee type configuration
4. Department configuration
5. Location configuration
6. Entity / Company configuration
7. Tenant default configuration
```

Example:

```text
Candidate: Rahul
Entity: ABC India
Department: Engineering
Employee Type: Full Time
Work Mode: Office

System checks:
- Is there a candidate-specific workflow?
- Is there Engineering + Full Time + Office workflow?
- Is there Full Time workflow?
- Is there ABC India workflow?
- If not, use Tenant Default workflow.
```

# 26. Candidate Submission Flow

When candidate submits form:

```text
Candidate submits form
        ↓
System saves submission
        ↓
System validates mandatory fields
        ↓
System creates verification task based on workflow
        ↓
HR / Payroll / Manager verifies
        ↓
If approved:
    Map data to employee draft profile
    Move candidate to next configured stage
    Send configured email
        ↓
If rejected:
    Send correction email
    Candidate updates form
    Verification starts again
```

# 27. Final Wiring Logic

This is how everything is connected:

```text
Add Candidate
    ↓
Select Entity + Job Details
    ↓
System finds Entity Configuration
    ↓
System gets Default / Custom Workflow
    ↓
Workflow gets Kanban Board
    ↓
Workflow stages define forms, documents, emails, checklist, approvals, verification
    ↓
Candidate moves through Kanban stages
    ↓
Each stage triggers configured actions
    ↓
Candidate submits forms/documents
    ↓
Verification happens
    ↓
Data maps to employee draft
    ↓
Joining day setup happens
    ↓
Candidate is converted to employee
```

# Final Rule

Your onboarding module should work like this:

```text
Settings define what is allowed.
Libraries create reusable items.
Entity Configuration selects defaults per company/entity.
Workflow Configuration wires everything step-wise.
Candidate page runs the actual onboarding.
Kanban board tracks progress.
Logs track everything.
```

This is the correct complete structure for your HRMS onboarding module.
